# CSV 契約: 交易匯入與匯出

**日期**: 2026-05-11  
**格式**: RFC 4180 相容 CSV，UTF-8，包含標題列。

## 欄位順序

CSV 必須使用固定六欄位，順序如下:

```csv
日期,類型,金額,分類,帳戶,備註
```

| 欄位 | 必填 | 格式 | 規則 |
|------|------|------|------|
| 日期 | 是 | `YYYY-MM-DD` | Asia/Taipei 本地交易日期，不得晚於今日。 |
| 類型 | 是 | `收入` 或 `支出` | 決定分類類型與帳戶餘額方向。 |
| 金額 | 是 | decimal text | > 0，最多 2 位小數，不含幣別符號與千分位逗號。 |
| 分類 | 是 | text | trim 後不可空白；不存在時依類型建立自訂分類。 |
| 帳戶 | 是 | text | trim 後必須符合既有帳戶 normalized name。 |
| 備註 | 否 | text | 可包含逗號、引號與換行；需安全清理。 |

## 匯入範例

```csv
日期,類型,金額,分類,帳戶,備註
2026-02-01,支出,150,餐飲,現金,"午餐, 便當"
2026-02-02,收入,50000,薪資,銀行,"二月薪資"
```

## 匯入規則

- 檔案大小必須小於 5 MB。
- 有效資料列不得超過 10,000 筆。
- 第一列必須為標題列，欄位名稱與順序必須完全符合。
- 空檔案或僅標題列回報「無有效資料」，不建立交易。
- 每列獨立驗證；成功列建立交易，失敗列跳過。
- 分類不存在時自動建立，分類類型由該列 `類型` 決定。
- 帳戶不存在時該列失敗，不自動建立帳戶。
- 日期、金額、分類、帳戶與注入風險都必須由伺服器端驗證。
- 批次建立交易與必要分類必須在一致性交易中完成；不可留下部分不一致資料。

## 匯入錯誤摘要

錯誤摘要必須包含:

- 成功筆數
- 失敗筆數
- 自動建立分類清單
- 每筆錯誤的 CSV 行號
- 欄位名稱
- 繁體中文原因

錯誤訊息範例:

- `第 4 行匯入失敗：日期無效`
- `第 7 行匯入失敗：金額必須大於 0`
- `第 9 行匯入失敗：帳戶不存在`
- `第 12 行匯入失敗：欄位數量不正確`

## 匯出規則

- 匯出未刪除交易。
- 未指定日期範圍時匯出全部未刪除交易。
- 日期範圍為包含起訖日期。
- 欄位順序與匯入相同。
- 日期輸出為 `YYYY-MM-DD`。
- 類型輸出為 `收入` 或 `支出`。
- 金額輸出使用 invariant decimal text，不輸出貨幣符號。
- 文字欄位必須正確處理逗號、引號、CRLF 與換行。

## 公式注入防護

匯出到試算表時，文字欄位若以以下字元開頭，必須採安全策略處理:

- `=`
- `+`
- `-`
- `@`
- tab
- CR
- LF

V1 採用在文字欄位前加單引號 `'` 的方式防止公式執行。加前綴僅用於 CSV 匯出內容，不修改資料庫原始備註。

## MIME 與檔名

- `Content-Type`: `text/csv; charset=utf-8`
- 建議檔名: `transactions-YYYYMMDD-HHmmss.csv`
- 下載回應不得快取敏感內容。

## 安全與隱私

- CSV 匯入不得把完整原始檔長期保存，除非後續 spec 明確要求。
- 匯入錯誤預覽需截斷或遮罩。
- 匯入匯出事件需記錄遮罩稽核摘要。
- 使用者資料不得因 CSV 功能傳送至第三方服務。
